# Eternity II engine — C / WebAssembly port

A from-scratch **C** re-implementation of the Eternity II engine, compiled to
**WebAssembly with plain `clang` (no emscripten)** as a freestanding `wasm32`
module. It is a byte-for-byte twin of the Rust engine in `engine/src/` and a
drop-in, build-time-switchable backend for the community website (it exposes the
exact surface of `web/src/engine/index.ts`).

The algorithm itself is documented once, language-agnostically, in
[`engine-side-quests/ALGORITHM.md`](../../../engine-side-quests/ALGORITHM.md).
`engine.c` is a literal translation of it and cross-references each section in
comments. Read the ALGORITHM doc first; this README only covers the C/WASM
specifics.

## Files

| File | What it is |
| --- | --- |
| `engine.c` | The whole engine: RNG, generator, official set, paths, step-able solver, scorer, and the WASM ABI. Also contains a `#ifndef __wasm__` `main` that emits golden-format output for native diffing. |
| `official_data.h` | The official 16×16 set, auto-generated from `engine/data/official_eternity2.csv` (binary color words decoded to decimal; hints as `{pos,piece,rot}`). Regenerated by `build.sh`. |
| `build.sh` | Builds `engine.wasm` with clang. Regenerates `official_data.h`, runs `wasm-opt` if present. `--native` also builds `engine_native`. |
| `glue.ts` | TypeScript loader + ABI wrapper. Exports the **same surface** as `web/src/engine/index.ts` (8 functions + `createSolver → SolverHandle`). Passes the project's strict typecheck/lint (no `any`/`!`/`@ts-*`). |
| `parity.mjs` | Runs `engine.wasm` under Node and checks every line of `engine-lua/golden.txt`. This is the authoritative parity gate. |
| `engine.wasm` | The committed build artifact, so the site builds on CI without a C toolchain. |

## Rebuilding

```sh
cd web/src/engine-c
./build.sh            # -> engine.wasm
./build.sh --native   # also -> engine_native (for golden diffing)
```

Apple's stock `clang` has **no `wasm32` backend**; `build.sh` auto-selects a
wasm-capable clang (Homebrew LLVM at `/opt/homebrew/opt/llvm/bin/clang`, or
`$CLANG`). Linking needs **`wasm-ld`** (the `lld` Homebrew formula:
`brew install lld`).

### Validate

```sh
node parity.mjs       # checks engine.wasm against golden.txt; exits non-zero on mismatch

# ABI-independent second check:
./build.sh --native
./engine_native | diff - ../../../engine-side-quests/engine-lua/golden.txt
```

## Freestanding wasm design

- Built with `clang --target=wasm32 -nostdlib -ffreestanding -O3
  -Wl,--no-entry -Wl,--export-dynamic`. **No libc.**
- **No allocator.** All working memory is *static, fixed-size buffers* sized for
  the official 16×16 board (256 pieces / 256 cells), which is the largest puzzle
  the site ever runs. This is the simplest, safest choice for a single-board
  engine and avoids a bump allocator entirely. There is exactly one live puzzle
  (`g_puzzle`) and one live solver (`g_solver`) at a time; the JS side
  serialises access (the website only runs one solver).
- The RNG needs 64-bit wrapping math — native in C via `unsigned long long`, no
  masking (unlike the JS/BigInt port). `typedef`s replace `<stdint.h>` since
  there is no libc under `-nostdlib`.

## The ABI (C ↔ JS)

WASM can't pass structs or strings across the boundary, so the C side exposes
primitive-only functions plus pointers into linear memory; `glue.ts` copies data
out with typed-array views. Key entry points (all `export_name`-annotated in
`engine.c`):

- **Puzzle builders** fill `g_puzzle` and return the piece count:
  `e2_official()`, `e2_generate(size,colors,seed)`,
  `e2_generate_solved(size,colors,seed)`.
- **Puzzle readers:** `e2_puzzle_width/height/colors/hints()`,
  `e2_pieces_ptr()` (→ `n*4` bytes, URDL), `e2_hint_pos/piece/rot(i)`.
- **Paths:** `e2_path_kind_count()`, `e2_path_kind_name(i)` (→ C string),
  `e2_build_path(kindIndex,w,h,seed)` (→ length, writes `g_path`),
  `e2_path_ptr()`.
- **Scoring:** write the board into `e2_board_in_ptr()` (`n` `i32`s), then
  `e2_score_board()`.
- **Solver** (single static instance): `e2_solver_new(pathLen, useHints,
  shufflePieces, seed)` (path must already be staged in `g_path`); then
  `e2_solver_step(budget)`, `e2_solver_status()`,
  `e2_solver_nodes/attempts/backtracks()` (returned as `double` — counters can
  exceed 2³²), `e2_solver_placed/best_placed()`, `e2_solver_board/best_board()`
  (→ `g_board_out`, `n` `i32`s), `e2_solver_score/best_score()`,
  `e2_solver_reset(...)`.

`glue.ts` maps the public names (`getOfficialPuzzle`, `createSolver`, …) onto
these. `SolverHandle.free()` is a no-op (no per-solver heap to release).

## Parity

`parity.mjs` is the gate. It instantiates `engine.wasm` and replays every
`GEN / OFF / OFFHINT / PATH / SOLVE / OFFICIALRUN` line from the Rust-produced
`engine-lua/golden.txt`, asserting field-for-field and **count-for-count**
(nodes / attempts / backtracks). Matching the counts proves the C engine walks
the identical search tree, not merely that it also solves.
